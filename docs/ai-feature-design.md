# Hiroe Tech Notes AI 機能 基本設計方針・記事要約生成仕様書

## 1. 概要

本ドキュメントは、Hiroe Tech Notes に AI を使った機能を追加する際の基本設計方針と、初期実装対象である「記事要約自動生成」機能の要件を定義する。

初期実装では、管理画面の記事作成・編集画面において、記事本文から要約を AI で生成し、記事の `excerpt` に反映できる補助機能を提供する。AI 生成結果は公開前に管理者が確認・修正する前提とし、自動公開や無確認保存は行わない。

## 2. AI 機能追加の基本設計方針

### 2.1 位置づけ

- AI 機能は、管理者の執筆・編集作業を補助する機能として扱う。
- AI の出力は確定情報ではなく、管理者が確認・編集するための下書きとして扱う。
- 公開サイトの閲覧者に対して、AI が直接応答する機能は初期方針の対象外とする。
- 記事本文、要約、タイトル、タグなど、公開コンテンツの最終責任は管理者にある。

### 2.2 明示操作

- AI 呼び出しは、管理者の明示操作によって実行する。
- 保存、公開、プレビューなどの既存操作に紐づけて暗黙的に AI を呼び出さない。
- AI を呼び出す UI には、外部 AI サービスに本文を送信することが分かる表現を添える。
- 生成結果は対象入力欄へ反映するが、記事保存は別操作とする。

### 2.3 外部サービス連携

- AI プロバイダー連携は、専用のサービスクラスに閉じ込める。
- コントローラー、ビュー、Stimulus controller から外部 API の URL や認証ヘッダーを直接組み立てない。
- API トークン、アカウント ID、モデル名、タイムアウト値などは Rails credentials または環境変数から読み込む。
- 認証情報、API レスポンス全文、本文全文をログに出力しない。
- 外部 API が失敗した場合でも、記事編集画面の既存操作を妨げない。

### 2.4 データ取り扱い

- AI に送信するデータは、目的達成に必要な最小限にする。
- 記事要約では、原則として本文、タイトル、生成条件のみを送信する。
- 管理者メールアドレス、セッション情報、内部 ID、未使用のサイト設定などは送信しない。
- 秘密情報、顧客情報、認証情報を含む記事を扱う可能性があるため、送信前に管理者が本文を確認できる画面上の明示操作を必須とする。
- 初期実装では、AI への送信内容と生成結果をアプリケーション DB に履歴保存しない。

### 2.5 コスト・無料枠の扱い

- AI 機能は無料枠または低コスト運用を優先して設計する。
- 料金、無料枠、レート制限は変更される可能性があるため、実装時と本番投入前に公式ドキュメントを再確認する。
- モデルは用途に対して十分な品質を満たす範囲で、軽量・低単価のものを優先する。
- 1 回のリクエストで送信する本文量と生成文字数に上限を設ける。
- API 失敗時に自動で連続リトライし、無料枠やレート制限を消費し続ける実装は避ける。

### 2.6 安全性・品質

- AI 出力はサーバー側で長さ、空文字、想定形式を検証してからクライアントに返す。
- HTML を生成させる用途では、既存の `MarkdownRenderer` のサニタイズ方針に従い、直接 HTML を信頼しない。
- プロンプトはユーザー入力と指示文を明確に分離する。
- AI が生成した文言をそのまま公開する前提にせず、管理者による確認・修正フローを維持する。
- ネットワークエラー、認証エラー、レート制限、プロバイダー障害時には、管理画面上に短いエラーを表示する。

### 2.7 テスト方針

- test 環境では外部 AI API に接続しない。
- AI クライアントはスタブ可能なサービスクラスとして実装する。
- 管理者認証、入力バリデーション、成功レスポンス、失敗レスポンスをテストする。
- プロンプト文字列の詳細一致に依存しすぎず、本文・タイトル・制約が含まれることを中心に確認する。
- 既存の統合テスト方針に合わせ、必要に応じて `test/integration/blog_flow_test.rb` に管理画面フローの確認を追加する。

## 3. 初期 AI プロバイダー方針

### 3.1 採用候補

初期実装の採用候補は Cloudflare Workers AI とする。

理由は以下のとおり。

- REST API で Rails アプリケーションから直接呼び出せる。
- Cloudflare アカウント ID と API token で認証できる。
- 無料枠があり、記事要約のような軽量用途と相性がよい。
- 既存要件で画像保存先として Cloudflare R2 を採用しており、運用上のサービス集約がしやすい。

### 3.2 公式情報の確認結果

調査日: 2026-07-01

Cloudflare Workers AI の公式ドキュメントでは、REST API から以下の形式でモデルを実行できる。

```text
POST https://api.cloudflare.com/client/v4/accounts/{ACCOUNT_ID}/ai/run/{MODEL}
Authorization: Bearer {API_TOKEN}
Content-Type: application/json
```

2026-07-01 時点の公式 Pricing ドキュメントでは、無料割り当ては 10,000 Neurons / day、超過分は Workers Paid plan で従量課金とされている。料金や無料枠は変わり得るため、金額をアプリケーション仕様に固定しない。

関連公式ドキュメント:

- Cloudflare Workers AI REST API: https://developers.cloudflare.com/workers-ai/get-started/rest-api/
- Cloudflare Workers AI Pricing: https://developers.cloudflare.com/workers-ai/platform/pricing/
- Cloudflare Workers AI Limits: https://developers.cloudflare.com/workers-ai/platform/limits/
- Cloudflare Workers AI Data usage: https://developers.cloudflare.com/workers-ai/platform/data-usage/

### 3.3 モデル選定

- 初期候補モデルは `@cf/meta/llama-3.2-1b-instruct` とする。
- 品質不足が明確な場合は、`@cf/meta/llama-3.2-3b-instruct` または同等の軽量 instruct モデルを候補にする。
- モデル名は設定値として切り替え可能にし、コードに固定しない。
- モデル変更時は、要約品質、コスト、レスポンス時間を確認する。

## 4. 記事要約自動生成機能 要件

### 4.1 目的

管理者が記事本文を書いた後、記事一覧や OGP で使用する短い要約文を AI で生成できるようにする。要約作成の手間を減らしつつ、最終的な表現は管理者が調整できる状態にする。

### 4.2 対象画面

- 管理記事作成画面
- 管理記事編集画面

既存の `docs/images/edit.png` を基準とし、記事本文欄と要約欄の関係が分かる位置に生成操作を追加する。

### 4.3 利用者

- ログイン済み管理者のみ利用できる。
- 未ログインユーザー、公開サイト閲覧者、通常の公開ページからは利用できない。

### 4.4 UI 要件

- 記事フォームの要約欄付近に `AIで要約生成` ボタンを表示する。
- ボタン押下時、現在フォームに入力されているタイトルと本文を使って要約を生成する。
- 生成中はボタンを disabled にし、処理中であることを表示する。
- 生成成功時は、生成された要約を要約欄へ反映する。
- 要約欄に既存入力がある場合は、上書き確認を行うか、明確に上書きされることが分かる UI にする。
- 生成失敗時は、要約欄を変更せず、管理画面内にエラーメッセージを表示する。
- 生成後も記事は自動保存しない。

### 4.5 入力要件

- AI 生成に使用する入力は、タイトルと本文とする。
- 本文が空の場合は AI を呼び出さず、本文入力を促すエラーを返す。
- 本文が長すぎる場合は、サーバー側で送信対象を上限文字数まで切り詰める。
- 初期の本文送信上限は 12,000 文字を目安とする。
- Markdown 記法はそのまま送信してよいが、添付画像 URL やコードブロックが多い場合でも要約本文に URL 羅列やコード詳細を含めないようプロンプトで制御する。

### 4.6 出力要件

- 生成する要約は日本語とする。
- 文字数は 70 から 90 字程度を目安とする。
- 1 段落のプレーンテキストとする。
- Markdown、HTML、箇条書き、引用符、前置き文を含めない。
- 記事本文にない事実を追加しない。
- 読者に内容が伝わる自然な文体にする。
- サーバー側で前後空白を除去し、最大 100 文字程度に収まるよう検証する。

### 4.7 プロンプト要件

プロンプトは、少なくとも以下の制約を含める。

- Hiroe Tech Notes の技術記事の要約であること。
- 日本語で 70 から 90 字程度に要約すること。
- 本文にない事実を追加しないこと。
- Markdown、HTML、箇条書き、見出しを使わないこと。
- 出力は要約本文のみとすること。

プロンプト本文には、タイトルと本文を明確な区切りで渡す。

### 4.8 サーバー側設計

実装候補は以下とする。

- AI クライアント: `CloudflareAiClient`
- 要約生成サービス: `PostSummaryGenerator`
- 管理画面エンドポイント: `Admin::PostSummariesController#create`
- ルーティング: `admin/posts` 配下、または `admin/post_summaries#create`

責務分離は以下とする。

- controller は認証、入力受付、HTTP ステータス返却を担当する。
- `PostSummaryGenerator` は本文整形、プロンプト構築、出力検証を担当する。
- `CloudflareAiClient` は Cloudflare Workers AI の HTTP 通信のみを担当する。
- モデル名、アカウント ID、API token は設定値から読み込む。

### 4.9 クライアント側設計

- Stimulus controller で `AIで要約生成` ボタンのクリックを処理する。
- 現在のフォーム値からタイトルと本文を取得し、管理画面エンドポイントへ JSON で送信する。
- CSRF token を付与する。
- 成功時はレスポンスの `summary` を要約 textarea に設定する。
- 失敗時はレスポンスの `error` を画面に表示する。
- importmap + Stimulus の既存構成を維持し、Node.js ツールチェーンは追加しない。

### 4.10 API レスポンス

成功時のレスポンス例:

```json
{
  "summary": "Terraform の状態管理を題材に、ローカル管理からリモートバックエンドへ移行する際の設計観点と運用上の注意点を整理した記事です。"
}
```

失敗時のレスポンス例:

```json
{
  "error": "本文を入力してから要約を生成してください。"
}
```

ステータスコードの目安:

- `200 OK`: 生成成功
- `400 Bad Request`: 本文未入力、入力上限超過など
- `401 Unauthorized` または `302 Found`: 未ログイン
- `429 Too Many Requests`: アプリ側または外部サービス側の制限
- `502 Bad Gateway`: Cloudflare Workers AI 呼び出し失敗

### 4.11 設定値

以下を Rails credentials または環境変数で設定できるようにする。

- `CLOUDFLARE_ACCOUNT_ID`
- `CLOUDFLARE_AI_API_TOKEN`
- `CLOUDFLARE_AI_MODEL`
- `CLOUDFLARE_AI_TIMEOUT_SECONDS`

初期値の考え方:

- モデル名は credentials または環境変数で明示する。
- タイムアウトは 10 秒程度を目安とする。
- API token が未設定の場合、ボタンを非表示にするか、押下時に設定不足エラーを表示する。

### 4.12 エラー・例外処理

- API token 未設定の場合は外部 API を呼び出さない。
- Cloudflare API が `success: false` を返した場合は失敗として扱う。
- レスポンス JSON が想定形式でない場合は失敗として扱う。
- タイムアウト時は失敗として扱い、管理者に再試行可能なエラーを表示する。
- ログにはエラー種別、HTTP ステータス、リクエスト ID などの診断情報のみを残し、本文やトークンは残さない。

### 4.13 レート制限

- 初期実装では、管理者操作に限定することで通常利用時の呼び出し回数を抑える。
- 必要に応じて、セッション単位または IP 単位で短時間の連続実行を制限する。
- ブラウザ側では生成中の多重押下を防止する。
- サーバー側でも同一リクエストの連続処理を避ける。

### 4.14 テスト要件

- 未ログインでは要約生成エンドポイントを利用できないこと。
- ログイン済み管理者が本文とタイトルを送ると要約が返ること。
- 本文が空の場合、外部 API を呼ばずにエラーを返すこと。
- AI クライアントが失敗した場合、要約欄を更新しないためのエラーを返すこと。
- API token や account ID が未設定の場合、分かりやすいエラーを返すこと。
- test 環境では Cloudflare Workers AI に実接続しないこと。

## 5. 非対象

- 公開記事を保存・公開するタイミングでの自動要約生成
- 生成履歴の保存・比較 UI
- 複数候補の同時生成
- タイトル生成、タグ生成、本文校正、翻訳
- 公開サイト閲覧者向けの AI チャット
- AI 生成結果で記事本文を自動変更する機能
- Cloudflare Workers や Queue を使った非同期生成

## 6. 将来拡張候補

- タイトル案生成
- タグ候補生成
- 記事本文の校正・読みやすさチェック
- OGP 説明文生成
- 複数モデルの切り替え
- AI Gateway による利用状況集計
- 生成履歴の保存と差分比較
- 長文記事向けの分割要約

## 7. 受け入れ条件

- 管理者が記事編集画面で `AIで要約生成` を押すと、本文から要約が生成され、要約欄に反映される。
- 要約生成後も記事は自動保存されない。
- 本文未入力時は AI API を呼ばず、画面上にエラーが表示される。
- Cloudflare 認証情報が未設定でもアプリ全体は起動し、通常の記事編集は継続できる。
- test 環境の自動テストは外部ネットワークに依存しない。
- API token、本文全文、AI レスポンス全文がログに出力されない。
