# Hiroe Tech Notes モバイル対応要件書

## 1. 概要

本書は、`docs/requirements.md` に定義された公開側機能を前提に、閲覧者がスマートフォンで閲覧する画面のモバイル向け差分要件を定義する。

参照画像は `docs/mobile_images` 以下の以下7画面とする。

- `docs/mobile_images/home.png`
- `docs/mobile_images/posts_index.png`
- `docs/mobile_images/experiments.png`
- `docs/mobile_images/article_show.png`
- `docs/mobile_images/tags.png`
- `docs/mobile_images/profile.png`
- `docs/mobile_images/about.png`

## 2. 対象範囲

### 2.1 対象画面

今回の対象は閲覧者が閲覧する公開側画面のみとする。

- ホーム
- 記事一覧
- 実験ログ一覧
- 記事詳細
- タグ一覧
- プロフィール
- About

### 2.2 対象外

以下は今回の対象外とし、本書では要件化しない。

- 管理画面
- 管理者ログイン画面
- 記事作成・編集画面
- 管理設定画面
- 管理用コメント一覧
- 管理者ログイン中だけに表示される編集導線のモバイル最適化

## 3. モバイル共通要件

### 3.1 基準 viewport

- 参照画像は 853px 幅のスマートフォン縦向き表示を基準とする。
- 実装時は 375px から 430px 程度の一般的なスマートフォン幅でも同じ情報設計が崩れないこと。
- 主要コンテンツは横スクロールを発生させず、本文、カード、フォーム、チップ、コードブロックが画面幅内に収まること。
- コードブロックなど横幅が不足するコンテンツのみ、ブロック内部で横スクロールを許容する。

### 3.2 ノート風背景

- デスクトップ要件と同じく、薄いアイボリーの紙質感と方眼罫を画面全体に表示する。
- 左端にはリングノートの穴またはリング表現を縦方向に表示する。
- モバイルではリング表現が本文を圧迫しないよう、本文の左余白を確保する。
- リング表現はスクロール中もページ左端に一貫して見えること。

### 3.3 レイアウト

- 公開側の全画面は、モバイルでは1カラム構成とする。
- デスクトップで右サイドバーに表示していたプロフィール、カテゴリー、アーカイブ、目次、関連記事、シェアなどは、モバイルでは本文の邪魔にならない位置へ移動するか、画面ごとの専用セクションに再構成する。
- 記事詳細では、目次を本文上部の折りたたみ可能なボックスとして表示し、右サイドバーは表示しない。
- ページ本文の左右余白は、リング表現を除いたコンテンツ領域内で読みやすい幅を保つ。

### 3.4 ヘッダー

- 画面上部に共通ヘッダーを表示する。
- ヘッダー左側にサイト名 `Tech Notes` を大きく表示する。
- サイト名の下にサブコピー `あるエンジニアの技術ノート` をオレンジで表示する。
- ヘッダー右側に以下のアイコン操作を表示する。
  - メニュー
  - テーマ切り替え
  - 検索
- アイコンはタップしやすいサイズと間隔を確保する。
- 画面によってアイコンの並び順が `メニュー / テーマ / 検索` または `テーマ / 検索 / メニュー` に見えるが、実装では全公開画面で一貫した順序に統一する。
- アイコンのみの操作にはアクセシブルな名前を付与する。

### 3.5 グローバルナビゲーション

- ヘッダー下に公開側グローバルナビゲーションを表示する。
- ナビゲーション項目は以下を基本とする。
  - ホーム
  - 記事一覧
  - 実験ログ
  - タグ一覧
  - プロフィール
  - About
- 現在表示中のナビゲーション項目は、オレンジの文字色またはオレンジの下線で強調する。
- 横幅が不足する場合でも、ナビゲーションの文字が折り返されて高さが不自然に増えないこと。
- 狭い幅では横スクロール、表示項目の圧縮、またはメニュー内への退避により、全項目へ到達できること。
- 参照画像の `home.png` では `About` が見えていないため、幅不足時に最後尾項目が画面外へ逃げる場合でも、操作不能にならないこと。

### 3.6 パンくず

- 下層ページでは、ナビゲーション下にパンくずを表示する。
- パンくずは `ホーム › 現在ページ` を基本とし、記事詳細では `ホーム › 記事一覧 › カテゴリー` を表示する。
- パンくずは本文より控えめな色とサイズにし、タップ可能な項目はリンクとして動作する。

### 3.7 見出し

- ページ見出しは大きく太い文字で表示する。
- 見出し下にはオレンジから薄い色へ抜ける手書き風の下線を表示する。
- 記事本文のセクション見出しは、左側のオレンジ罫線付きで表示する。

### 3.8 ボタン、チップ、カード

- 主要ボタンはオレンジ背景、白文字とする。
- 副次ボタンは白背景、薄い枠線、黒文字とする。
- カテゴリー、タグ、ページネーションはチップまたは小さなボタンとして表示する。
- チップは複数行に折り返せること。
- カードは薄い枠線、控えめな角丸、紙面に馴染む背景で表示する。
- タップ対象は最低44px程度の高さを確保する。

### 3.9 文字量と可読性

- 本文はスマートフォンで読みやすい行間を確保する。
- 長いタイトルやタグ名はカード外にはみ出さず、必要に応じて折り返す。
- 日本語本文の不自然な単語分断を避ける。
- 英数字やURLなどの長い連続文字列は、画面幅内で折り返すか、対象ブロック内でスクロールさせる。

## 4. ホーム画面

参照画像: `docs/mobile_images/home.png`

### 4.1 ヒーロー

- ヘッダーとナビゲーションの直下にヒーローエリアを表示する。
- ヒーロー見出しとして `コードとインフラを、動かしながら学ぶ。` を大きく表示する。
- 補足文として、インフラとコードの間で学んだことや試したことをノートに残している旨を表示する。
- 特徴を3項目表示する。
  - クラウド / インフラ / 自動化が中心
  - 小さな実験と、そこから得た学びを記録
  - 誰かのヒントになればうれしいです
- 特徴の各項目にはオレンジのチェックアイコンを表示する。
- デスクトップで右側に配置される要素は、モバイルでは縦方向に続けて配置する。

### 4.2 今日のひとことログ

- ヒーロー下に `# 今日のひとことログ` カードを表示する。
- カード右上にクリップ風装飾を表示する。
- カードには以下を表示する。
  - 日付
  - タイトル
  - コードスニペット
  - メモ
  - 実験ログへのリンク
- コードスニペットは黒背景で表示し、スマートフォン幅内に収まること。
- `実験ログに続く →` リンクをカード右下に表示する。

### 4.3 最新の記事

- `最新の記事` セクションを表示する。
- セクション見出しの左にオレンジ罫線を表示する。
- カテゴリー絞り込みチップを表示する。
  - すべて
  - インフラ
  - AWS
  - 自動化
  - プログラミング
  - セキュリティ
  - AI開発
- 選択中のチップはオレンジ背景で表示する。
- 記事はコンパクトなリスト形式で表示する。
- 各記事には以下を表示する。
  - 日付
  - タイトル
  - タグ
  - 詳細へ進むことを示す右向きアイコン
- リスト下部に `記事一覧へ →` リンクを表示する。

### 4.4 ホームで表示しない要素

- 参照画像では、デスクトップの右サイドバーに相当するプロフィールカード、カテゴリー一覧、アーカイブ一覧はホームの表示範囲に出ていない。
- モバイルのホームでは、ファーストビューから最新記事までの流れを優先し、サイドバー要素を上部に割り込ませない。

## 5. 記事一覧画面

参照画像: `docs/mobile_images/posts_index.png`

### 5.1 ページ構成

- パンくず `ホーム › 記事一覧` を表示する。
- ページ見出し `記事一覧` を表示する。
- 見出し下にオレンジの手書き風下線を表示する。

### 5.2 検索と並び替え

- 見出し下に検索入力を表示する。
- 検索入力には検索アイコンとプレースホルダー `タイトルで検索` を表示する。
- 検索入力の下に並び順セレクトを表示する。
- 並び順の初期表示は `新しい順` とする。
- 検索ボタンとリセットボタンを横並びで表示する。
- 検索ボタンはオレンジ背景、リセットボタンは白背景の枠線ボタンとする。

### 5.3 カテゴリー絞り込み

- 検索操作の下にカテゴリー絞り込みチップを表示する。
- チップは複数行に折り返して表示する。
- 表示項目は以下を基本とする。
  - すべて
  - インフラ
  - AWS
  - Azure
  - 自動化
  - プログラミング
  - セキュリティ
- 選択中のチップはオレンジ背景で表示する。

### 5.4 記事カード

- 記事はカード形式で縦に並べる。
- 各カード左上に記事アイコンを表示する。
- 各カードには以下を表示する。
  - 公開日
  - カテゴリー
  - 読了時間
  - タイトル
  - 要約
  - タグ
- カテゴリーはオレンジ文字で表示する。
- タイトルは太字で2行程度まで自然に折り返す。
- カード全体またはタイトルをタップすると記事詳細へ遷移する。

### 5.5 ページネーション

- 一覧下部にページネーションを表示する。
- `前へ`、現在ページ番号、`次へ` を横並びで表示する。
- 現在ページ番号はオレンジ背景で表示する。

### 6.1 ページ構成

## 7. 記事詳細画面

参照画像: `docs/mobile_images/article_show.png`

### 7.1 ページ構成

- パンくず `ホーム › 記事一覧 › カテゴリー` を表示する。
- 記事ヘッダー、目次、本文、コメントの順に1カラムで表示する。
- デスクトップ右サイドバーはモバイルでは表示しない。

### 7.2 記事ヘッダー

- 記事タイトル上にカテゴリーまたはタグのチップを表示する。
- 記事タイトルを大きく太い文字で表示する。
- 長いタイトルは画面幅内で自然に複数行へ折り返す。
- メタ情報をアイコン付きで表示する。
  - 公開日
  - 更新日
  - 読了時間
  - 著者
- タグをチップ形式で表示する。
- メタ情報とタグは、画面幅に応じて複数行に折り返す。
- `docs/requirements.md` では記事ヘッダーに要約文を表示するが、モバイルでは参照画像に合わせ、記事本文への導入を短く保つため要約文を必須表示しない。
- 要約文を表示する場合は、タイトル、メタ情報、タグ、目次の流れを妨げない短い段落として配置する。

### 7.3 目次

- 記事本文の前に目次ボックスを表示する。
- 目次ボックスにはリストアイコン、見出し `目次`、開閉アイコンを表示する。
- 目次には記事内見出しを番号付きで表示する。
- 目次項目をタップすると該当見出しへ移動する。
- 目次は必要に応じて折りたたみ可能とし、開閉状態が支援技術に伝わること。

### 7.4 本文

- 本文は読みやすい行間と余白で表示する。
- セクション見出しには左側のオレンジ罫線を表示する。
- 注意枠は薄い背景、枠線、アイコン付きで表示する。
- コードブロックは黒背景で表示し、右上にコピーアイコンを表示する。
- コードブロックは画面幅を超える場合、ブロック内で横スクロールできること。
- Markdown画像の拡大表示要件は `docs/requirements.md` の記事詳細ページ要件に従い、モバイルでも画面内に収まるオーバーレイとして表示する。

### 7.5 コメント

- 記事本文の下部にコメントセクションを表示する。
- セクション見出し `コメント` をアイコン付きで表示する。
- コメントの説明文またはコメント一覧、投稿フォームは `docs/requirements.md` のコメント機能要件に従う。
- モバイルでは入力欄、Turnstile、投稿ボタンが画面幅内に収まり、横スクロールしないこと。

### 8.1 ページ構成

- パンくず `ホーム › About` を表示する。

- 本文は大きめの文字と広めの行間で表示する。

- 説明本文の下に、3項目のコンセプトカードを表示する。
- 各項目にはオレンジの丸枠アイコンを表示する。
- 表示項目は以下を基本とする。

- 画像とテキストはモバイル幅で読みやすい配置にし、必要に応じて縦積みに切り替える。

## 11. 検索・テーマ・メニュー操作

### 11.1 検索

- ヘッダーの検索アイコンから検索機能へアクセスできること。
- 検索UIをモーダル、ドロワー、検索ページのいずれで提供する場合も、スマートフォン幅で入力欄と結果が画面外へはみ出さないこと。
- 検索アイコンのタップ対象にはアクセシブルな名前を付与する。

### 11.2 テーマ切り替え

- ヘッダーのテーマ切り替えアイコンからテーマを切り替えられること。
- 初期実装ではライトテーマを完成対象とし、ダークテーマは将来拡張できる構造にする。
- テーマ切り替えアイコンは現在状態が支援技術に伝わること。

### 11.3 メニュー

- ヘッダーのメニューアイコンは、狭い画面でナビゲーションまたは補助導線を開くための操作として使用できること。
- グローバルナビゲーションが画面上に全項目表示される場合でも、メニューアイコンは将来のドロワー拡張を妨げない構造にする。
- メニューを開閉する場合は、開閉状態が支援技術に伝わり、Escキーまたは閉じる操作で閉じられること。

## 12. アクセシビリティ要件

- すべてのリンク、ボタン、フォーム部品はキーボードで操作できること。
- アイコンのみの操作にはアクセシブルな名前を付与すること。
- 現在ページのナビゲーション項目には `aria-current` などで現在位置を伝えること。
- 折りたたみ要素には開閉状態を伝える属性を付与すること。
- 色だけで状態を表現せず、下線、文字、アイコン、属性でも状態を判別できること。
- フォーム入力にはラベルまたはアクセシブルな名前を関連付けること。

## 13. パフォーマンス要件

- モバイルの初回表示を重くしないよう、プロフィール画像や記事画像は適切なサイズを配信する。
- ノート背景や装飾は過度に大きな画像に依存しない。
- ファーストビューに不要な管理画面用アセットを読み込まない。
