機能追加・機能修正の依頼を受けた場合、このスキルの手順に従うこと。

**各フェーズにおいて、依頼内容・要件・設計の意図などで不明瞭な点があれば、その時点でユーザーに質問し、明確にしてから次に進むこと。** 推測で進めると手戻りが発生するため、不明点は必ず確認する。

## 役割分担

- **オーケストレータ（このスキルを実行するエージェント）**: フェーズ1〜7（要件整理・設計・チケット分割）、タスクエージェントへの委譲と検収、全体チェック、コミット・PR作成を担当する。
- **タスクエージェント（task ツールで起動）**: 割り当てられたチケット1件の実装を、小ループで完遂する。コミット・プッシュ・ブランチ操作は行わない。

## ブランチ運用フロー

1. **ブランチ作成**: フェーズ7（チケット分割）の完了・ユーザー承認後、実装開始直前に `main` ブランチからトピックブランチを作成する (`git checkout -b <ブランチ名>`)。
2. **実装〜全体チェック**: フェーズ8〜10を実行する。タスクエージェントはコミットしない。
3. **コミット**: 全チェック通過後、オーケストレータが機能の論理単位でまとめてコミットする。
4. **プッシュ**: `git push origin <ブランチ名>` でリモートにプッシュする。
5. **PR作成**: `gh pr create --base main --head <ブランチ名> --title "..." --body "..."` でプルリクエストを作成する。
6. **レビュー依頼**: PRのURLをユーザーに伝え、レビューを依頼する。ユーザーがGitHub上でレビューし、マージする。
7. **マージ確認と後片付け**: ユーザーから「マージした」旨のメッセージを受けたら、`main` ブランチに切り替え、`git pull origin main` してマージを確認し、開発用ブランチを削除する (`git branch -d <ブランチ名>`)。

## ワークフロー概要

フェーズ1〜7は**各フェーズの成果物がまとまった時点でユーザーに確認を求め、了承を得てから次のフェーズに進むこと**。フェーズ8〜10はユーザー確認なしで自律的に大ループを回してよい。

```
背景・目的の明確化
  ↓
理想と現実の検討 → 最終案の決定
  ↓
要件の洗い出し
  ↓
現状調査（コード・ライブラリ・既存UI/UX）
  ↓
設計方針 → インターフェイス設計 → 内部設計 → DB設計
  ↓
チケット分割・実行計画の作成
  ↓
ブランチ作成
  ↓
タスクエージェントへ委譲（依存関係に基づき並列/直列を制御）
  ↓  各エージェント内の小ループ:
  ↓  実装 → 静的解析 → 関連テスト → セルフレビュー → 動作確認 ↺
  ↓
完了報告の検収（全チケット）
  ↓
全体チェック: 型チェック → Rubocop → セキュリティスキャン → 全テスト
  → システムテスト → コードレビュー → 機密情報スキャン
  ↓  問題があれば修正チケットを起票して委譲へ戻る（大ループ）
要件充足の確認
  ↓  未充足ならチケットを起票して委譲へ戻る（大ループ）
コミット → プッシュ → PR作成 → 報告
```

## フェーズ1: 背景・目的の明確化

依頼の背景と目的をユーザーに確認し、以下の項目を整理する。

- **背景**: なぜこの機能が必要か、何が課題か
- **目的**: この機能で何を達成したいか
- **スコープ**: やること・やらないことの境界
- **成功条件**: どうなったら完了と言えるかの明確な基準

成果物: 上記を箇条書きで一文ずつまとめたもの（ユーザーに確認すること）。不明点があれば確認してから次に進むこと。

---

## フェーズ2: 理想と現実の検討 → 最終案の決定

### 2.1 理想の洗い出し

実装制約を無視した理想の動作・UIを列挙する。

### 2.2 現実的な制約の確認

以下の観点から制約を洗い出す:

- **技術的制約**: 現在のスタック（Rails 8, Ruby 3.4, importmap + Stimulus, SQLite/PostgreSQL）で実現可能か
- **ライブラリ制約**: 追加ライブラリが必要か、既存ライブラリで代替可能か
- **期間制約**: おおまかな工数感
- **運用制約**: 既存機能との互換性、後方互換性、マイグレーションの要否

### 2.3 最終案の決定

理想と制約をすり合わせ、実装方針の最終案をユーザーに提案し、確認を得る。

成果物: 最終案の説明（箇条書きまたは短い文章、ユーザー確認済み）。案に確信が持てない場合は複数案を提示し、ユーザーに選んでもらうこと。

---

## フェーズ3: 要件の洗い出し

最終案を基に、実装レベルの要件を列挙する。

- 機能要件（何ができるべきか）
- UI要件（画面のどこに何が表示されるか、操作手順）
- 非機能要件（パフォーマンス、セキュリティ、アクセシビリティ）
- エラーハンドリング要件（異常系の振る舞い）
- テスト要件（どのようなテストケースが必要か）

各要件は `AGENTS.md` の記述ルールに従い日本語で記述すること。

要件は `docs/requirements.md` の該当セクションに追記すること（UI変更がある場合は特に）。また画面デザインの参考画像が `docs/images/*.png` にある場合は必ず参照すること。

成果物: 要件リスト（ユーザー確認済み） + `docs/requirements.md` への追記。要件の解釈に迷ったらユーザーに確認すること。

---

## フェーズ4: 現状調査

実装を始める前に、関連するコードの現状を把握する。

### 4.1 コード調査

- 追加・修正対象の機能に関連するファイルを特定する
- 類似機能の実装パターンを確認する（コントローラ、ビュー、モデル、ヘルパー、サービスのそれぞれ）
- 既存のテストを確認し、テストパターンを把握する

### 4.2 ライブラリ調査

- 今回の修正に使える既存ライブラリがあるか確認する
- 新規ライブラリ導入が必要な場合、Gemfile / importmap の管理ポリシーに合致するか確認する
- 代替案（既存ライブラリの機能で実現できないか）を検討する
- **外部ライブラリを追加する場合は、GitHub リポジトリで以下の観点を確認し、メンテナンスが継続されているものだけを選択すること:**
  - 最終コミットから1年以上経過していないか
  -  Issue や Pull Request が放置されていないか
  - 対応している Ruby / Rails のバージョンが現在のスタックと整合しているか
  - スター数やフォーク数である程度使われているか

### 4.3 UI/UX 一貫性調査

- 既存の同種画面・コンポーネントのデザインパターンを確認する
- 新しいコンポーネントを追加する場合、既存のコンポーネントの書き方・命名規則・レイアウトを確認する
- `docs/requirements.md` のデザイン要件セクションおよび `docs/images/*.png` のスクリーンショットを参照する
- テーマ（light/dark）対応が必要な場合、既存のCSS変数とクラス分岐パターンを確認する

成果物: 調査結果の整理（ファイルパス・使用ライブラリ・既存パターンの引用を明記）

---

## フェーズ5: 設計方針

調査結果を基に設計方針を定める。

- **アーキテクチャ上の配置**: どのレイヤーに何を追加するか（コントローラ、モデル、ビュー、ヘルパー、サービス、JavaScript、CSS）
- **既存パターンとの整合**: 類似機能と同じパターンに従うか、新しいパターンを導入するか
- **命名規則**: クラス名、メソッド名、ファイル名、ルートパラメータ、DBカラム名の命名方針
- **エラー設計**: エラーの種類とハンドリング方法

成果物: 設計方針のドキュメント（ユーザーに確認すること）。トレードオフがある場合は理由を含めて提示し、判断を仰ぐこと。

---

## フェーズ6: 詳細設計

### 6.1 インターフェイス設計

- コントローラのアクションとルーティング
- ビューで渡すインスタンス変数
- ヘルパーメソッドのシグネチャ
- サービスオブジェクトのパブリックメソッド
- JavaScript のイベントとStimulusコントローラのAPI
- フォームの入力項目とバリデーション

### 6.2 内部設計

- モデルの関連・バリデーション・コールバック
- サービスオブジェクトの内部ロジック
- 非同期処理の有無と方法
- 状態遷移（該当する場合）

### 6.3 DB設計

- 新しいテーブル・カラムの追加仕様
- インデックス
- マイグレーションの前後でのデータ整合性
- SQLite / PostgreSQL 両アダプタで動作することの確認

成果物: 設計ドキュメント（ユーザーに確認すること）。判断に迷う選択肢があれば、理由付きで提示して選んでもらうこと。

---

## フェーズ7: チケット分割・実行計画

詳細設計をもとに、実装作業をタスクエージェントへ委譲可能な単位のチケットに分割する。

### 7.1 チケットの作成

各チケットには以下を含めること:

- **タイトル**: 内容が一目で分かる名前
- **目的**: 何を実現するか
- **スコープ**: 対象ファイル・レイヤー、やること・やらないこと
- **依存チケット**: 先行完了が必要なチケット（あれば）
- **完了条件**: 追加・更新するテスト、期待される振る舞い、通過すべきチェック

粒度の目安:

- 1チケット = 1エージェントが1コンテキストで完結できる大きさ
- チケット数は2〜10程度を目安とし、大きすぎるチケットはさらに分割する
- 1行修正レベルの細かすぎる分割は避ける

### 7.2 依存関係の分析と実行計画

- チケット間の依存関係（DBスキーマ → モデル → コントローラ/ビュー、共通ヘルパー → 利用側など）を洗い出す
- 依存のないチケットは並列グループ、依存のあるチケットは直列の順序付けを行い、実行計画を作成する
- **並列グループ内のチケットは変更ファイルが重複しないようにすること**。重複が避けられない場合は同一チケットに統合するか直列実行とする
  - 注意: 本リポジトリは `test/integration/blog_flow_test.rb` にアサーションが集中するため、複数チケットがこのファイルに追記する場合は特に競合しやすい

成果物: チケット一覧 + 実行計画（並列/直列のグループ分け）。会話内で提示し、ユーザーの承認を得ること。ファイルには保存しない。

---

## フェーズ8: ブランチ作成・タスクエージェント実行

### 8.1 ブランチ作成

`main` の最新から作成する:

```bash
git checkout main && git pull origin main
git checkout -b <ブランチ名>
```

### 8.2 タスクエージェントへの委譲

- 実行計画に従い、task ツールでタスクエージェントを起動する。並列グループは同一メッセージで同時に起動し、直列チケットは先行チケットの検収完了を待ってから起動する
- 各エージェントへのプロンプトには最低限以下を含めること:
  - チケットの内容（目的・スコープ・完了条件）
  - 設計方針・詳細設計のうち該当部分
  - 関連ファイルのパスと既存パターンの説明（フェーズ4の調査結果）
  - 制約:
    - コミット・プッシュ・ブランチ操作を行わないこと
    - チケットのスコープ外のファイルを変更しないこと（必要になった場合は報告して指示を仰ぐこと）
    - コードコメントはユーザーから要求された場合のみ追加すること
    - コミュニケーション・完了報告は日本語で行うこと
  - 8.3 の小ループ手順
  - 8.4 の完了報告フォーマット

### 8.3 小ループ（各タスクエージェントが遵守する手順）

1. **実装**: チケットのスコープ内でコードを書く
2. **静的解析**: 変更ファイルに対して `bin/rubocop <変更ファイル>` を実行。`# typed: true` 以上のファイルを変更した場合は `bundle exec srb tc` も実行
3. **関連テスト**: チケットで追加・変更した機能に関係するテストだけを実行する（例: `bin/rails test test/integration/blog_flow_test.rb -n test_xxx`）。**全量テストは大ループで実行するため、小ループでは実行しない**（並列実行中のテストDB競合を避けるため）
4. **セルフコードレビュー**: フェーズ9.6と同じ観点で自分の差分をレビューする
5. **動作確認**: 完了条件を満たすことを確認する（テスト結果に加え、必要に応じて `rails console` 等での検証）
6. 問題があれば 1 に戻る。完了条件をすべて満たしたら完了報告をして終了する

### 8.4 完了報告フォーマット

- 実施した内容の概要
- 追加・変更したファイルのリスト
- 実行したチェックとテストの結果
- 残課題・懸念事項（あれば）

### 8.5 検収（オーケストレータ）

- 報告内容がチケットの完了条件をすべて満たすか確認する。必要に応じて差分や関連テストの実行結果を自分でも確認する
- 不十分な場合は不足点を明示して修正を依頼する（同じエージェントへの追加指示、または新規エージェントの起動）
- 全チケットを検収したらフェーズ9へ進む

---

## フェーズ9: 全体チェック（大ループ）

全チケット検収後、オーケストレータが変更全体に対して以下を実行する。

### 9.1 型チェック（Sorbet）

```bash
bundle exec srb tc
```

### 9.2 構文・スタイルチェック（Rubocop）

```bash
bin/rubocop
```

### 9.3 セキュリティスキャン

```bash
bin/brakeman --no-pager
bin/bundler-audit
```

### 9.4 全テスト

```bash
bin/rails test
```

### 9.5 システムテスト

```bash
bin/rails test:system
```

UI・画面挙動の変更がある場合のみ実行する。

### 9.6 コードレビュー（変更全体）

チケット単位ではなく変更全体を通して、以下の観点で見直す。確認すべき点が不明な場合はユーザーに判断を仰ぐこと:

- **要件充足**: 全ての要件を満たしているか
- **エッジケース**: 空データ、異常値、認証なし、権限不足のケースを網羅しているか
- **エラーハンドリング**: ユーザーに適切なエラーメッセージが表示されるか
- **N+1**: データベースクエリにN+1が発生していないか
- **セキュリティ**: SQLインジェクション、XSS、CSRFの対策ができているか
- **チケット間の統合**: 重複コード、命名揺れ、インターフェイスの不一致、チケット間で矛盾した実装がないか
- **既存コードとの一貫性**: 同じ問題に対して同じパターンで解決しているか
- **CSSクラス名**: 既存の命名規則に従っているか
- **JavaScript**: Stimulus コントローラの規則に従っているか

### 9.7 機密情報スキャン

`.agents/skills/security-check/SKILL.md` の手順に従い、リポジトリ全体の機密情報スキャンを実行する。

- 🔴 CRITICAL または 🟠 HIGH の指摘が見つかった場合は作業を中断し、ユーザーに報告して指示を仰ぐこと
- 🟡 MEDIUM 以下の指摘のみの場合は報告を行うが、ユーザーの判断で継続してよい

### 9.8 問題発見時の対応

いずれかのチェックで問題が見つかった場合、修正内容をチケット化（完了条件を明記）し、フェーズ8へ戻って大ループを回す。修正チケットの規模が小さく依存もなければ、直列1件のチケットとして1エージェントに委譲する。

---

## フェーズ10: 要件充足の確認

フェーズ3の要件リストと突き合わせ、未充足の要件がないか確認する。

- 未充足の要件がある場合はチケットを起票し、フェーズ8へ戻って大ループを回す
- 全要件を充足し、フェーズ9の全チェックを通過したらフェーズ11へ進む

---

## フェーズ11: コミット・報告・PR作成

1. `git status` / `git diff` で変更内容を最終確認し、意図しない変更が混入していないか確認する
2. 機能の論理単位でコミットを分割してコミットする（日本語メッセージ）

```bash
git add <該当ファイル>
git commit -m "{日本語のコミットメッセージ}"
git push origin <ブランチ名>
```

3. PR を作成する

```bash
gh pr create --base main --head <ブランチ名> --title "..." --body "..."
```

4. 以下の内容をユーザーに報告する:

- 実施した内容の概要
- 消化したチケットの一覧
- 追加したファイルのリスト
- 変更したファイルのリスト
- （必要な場合）動作確認方法や注意点
- PR のURL（レビューを依頼する）

5. ユーザーから「マージした」旨のメッセージを受けたら、ブランチ運用フローの手順で後片付けを行う
