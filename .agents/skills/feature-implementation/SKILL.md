description: 機能追加・機能修正の全工程を定義する。背景整理から設計・実装・テスト・コードレビュー・セキュリティチェックまでをカバーする。
---

機能追加・機能修正の依頼を受けた場合、このスキルの手順に従うこと。

**各フェーズにおいて、依頼内容・要件・設計の意図などで不明瞭な点があれば、その時点でユーザーに質問し、明確にしてから次に進むこと。** 推測で進めると手戻りが発生するため、不明点は必ず確認する。

## ワークフロー概要

以下の各フェーズを順に実行する。**各フェーズの成果物がまとまった時点でユーザーに確認を求め、了承を得てから次のフェーズに進むこと。** 要件を完全に満たせていない場合は実装フェーズに戻ってループする。

```
背景・目的の明確化
  ↓
理想と現実の検討 → 最終案の決定
  ↓
要件の洗い出し
  ↓
現状調査（コード・ライブラリ・既存UI/UX）
  ↓
設計方針 → インターフェイス設計 → 内部設計 → DB設計
  ↓
実装
  ↓
Rubocop + ユニットテスト + システムテスト
  ↓
コードレビュー
  ↓
セキュリティチェック
  ↓
条件を満たすまでループ
  ↓
コミット・報告
```

---

## フェーズ1: 背景・目的の明確化

依頼の背景と目的をユーザーに確認し、以下の項目を整理する。

- **背景**: なぜこの機能が必要か、何が課題か
- **目的**: この機能で何を達成したいか
- **スコープ**: やること・やらないことの境界
- **成功条件**: どうなったら完了と言えるかの明確な基準

成果物: 上記を箇条書きで一文ずつまとめたもの（ユーザーに確認すること）。不明点があれば確認してから次に進むこと。

---

## フェーズ2: 理想と現実の検討 → 最終案の決定

### 2.1 理想の洗い出し

実装制約を無視した理想の動作・UIを列挙する。

### 2.2 現実的な制約の確認

以下の観点から制約を洗い出す:

- **技術的制約**: 現在のスタック（Rails 8, Ruby 3.4, importmap + Stimulus, SQLite/PostgreSQL）で実現可能か
- **ライブラリ制約**: 追加ライブラリが必要か、既存ライブラリで代替可能か
- **期間制約**: おおまかな工数感
- **運用制約**: 既存機能との互換性、後方互換性、マイグレーションの要否

### 2.3 最終案の決定

理想と制約をすり合わせ、実装方針の最終案をユーザーに提案し、確認を得る。

成果物: 最終案の説明（箇条書きまたは短い文章、ユーザー確認済み）。案に確信が持てない場合は複数案を提示し、ユーザーに選んでもらうこと。

---

## フェーズ3: 要件の洗い出し

最終案を基に、実装レベルの要件を列挙する。

- 機能要件（何ができるべきか）
- UI要件（画面のどこに何が表示されるか、操作手順）
- 非機能要件（パフォーマンス、セキュリティ、アクセシビリティ）
- エラーハンドリング要件（異常系の振る舞い）
- テスト要件（どのようなテストケースが必要か）

各要件は `AGENTS.md` の記述ルールに従い日本語で記述すること。

要件は `docs/requirements.md` の該当セクションに追記すること（UI変更がある場合は特に）。また画面デザインの参考画像が `docs/images/*.png` にある場合は必ず参照すること。

成果物: 要件リスト（ユーザー確認済み） + `docs/requirements.md` への追記。要件の解釈に迷ったらユーザーに確認すること。

---

## フェーズ4: 現状調査

実装を始める前に、関連するコードの現状を把握する。

### 4.1 コード調査

- 追加・修正対象の機能に関連するファイルを特定する
- 類似機能の実装パターンを確認する（コントローラ、ビュー、モデル、ヘルパー、サービスのそれぞれ）
- 既存のテストを確認し、テストパターンを把握する

### 4.2 ライブラリ調査

- 今回の修正に使える既存ライブラリがあるか確認する
- 新規ライブラリ導入が必要な場合、Gemfile / importmap の管理ポリシーに合致するか確認する
- 代替案（既存ライブラリの機能で実現できないか）を検討する

### 4.3 UI/UX 一貫性調査

- 既存の同種画面・コンポーネントのデザインパターンを確認する
- 新しいコンポーネントを追加する場合、既存のコンポーネントの書き方・命名規則・レイアウトを確認する
- `docs/requirements.md` のデザイン要件セクションおよび `docs/images/*.png` のスクリーンショットを参照する
- テーマ（light/dark）対応が必要な場合、既存のCSS変数とクラス分岐パターンを確認する

成果物: 調査結果の整理（ファイルパス・使用ライブラリ・既存パターンの引用を明記）

---

## フェーズ5: 設計方針

調査結果を基に設計方針を定める。

- **アーキテクチャ上の配置**: どのレイヤーに何を追加するか（コントローラ、モデル、ビュー、ヘルパー、サービス、JavaScript、CSS）
- **既存パターンとの整合**: 類似機能と同じパターンに従うか、新しいパターンを導入するか
- **命名規則**: クラス名、メソッド名、ファイル名、ルートパラメータ、DBカラム名の命名方針
- **エラー設計**: エラーの種類とハンドリング方法

成果物: 設計方針のドキュメント（ユーザーに確認すること）。トレードオフがある場合は理由を含めて提示し、判断を仰ぐこと。

---

## フェーズ6: 詳細設計

### 6.1 インターフェイス設計

- コントローラのアクションとルーティング
- ビューで渡すインスタンス変数
- ヘルパーメソッドのシグネチャ
- サービスオブジェクトのパブリックメソッド
- JavaScript のイベントとStimulusコントローラのAPI
- フォームの入力項目とバリデーション

### 6.2 内部設計

- モデルの関連・バリデーション・コールバック
- サービスオブジェクトの内部ロジック
- 非同期処理の有無と方法
- 状態遷移（該当する場合）

### 6.3 DB設計

- 新しいテーブル・カラムの追加仕様
- インデックス
- マイグレーションの前後でのデータ整合性
- SQLite / PostgreSQL 両アダプタで動作することの確認

成果物: 設計ドキュメント（ユーザーに確認すること）。判断に迷う選択肢があれば、理由付きで提示して選んでもらうこと。

---

## フェーズ7: 実装

設計に従って実装する。

- 実装中に設計の解釈や実装方法に不明点が生じた場合は、推測せずにユーザーに質問すること
- コミットは機能の単位で分割する（1機能 = 1コミットが理想）
- コミットメッセージは日本語で記述する
- 実装中に設計の見直しが必要になった場合はユーザーに相談する
- コードコメントは不要（ユーザーから要求があった場合のみ追加）

---

## フェーズ8: 静的解析・テスト

### 8.1 Rubocop

```bash
bin/rubocop
```

指摘がある場合は修正すること。

### 8.2 ユニットテスト / 統合テスト

```bash
bin/rails test
```

新規テストを追加し、既存テストが全てパスすることを確認する。

### 8.3 システムテスト

```bash
bin/rails test:system
```

ブラウザ駆動のテストが必要な変更の場合のみ実行する。新規システムテストが必要な場合は `test/system/` に追加する。

---

## フェーズ9: コードレビュー

自分自身でコードレビューを行う。以下の観点で見直す。確認すべき点が不明な場合はユーザーに判断を仰ぐこと:

- **要件充足**: 全ての要件を満たしているか
- **エッジケース**: 空データ、異常値、認証なし、権限不足のケースを網羅しているか
- **エラーハンドリング**: ユーザーに適切なエラーメッセージが表示されるか
- **N+1**: データベースクエリにN+1が発生していないか
- **セキュリティ**: SQLインジェクション、XSS、CSRFの対策ができているか
- **既存コードとの一貫性**: 同じ問題に対して同じパターンで解決しているか
- **CSSクラス名**: 既存の命名規則に従っているか
- **JavaScript**: Stimulus コントローラの規則に従っているか

---

## フェーズ10: セキュリティチェック

`git commit` を実行する前に必ず `.agents/skills/security-check/SKILL.md` の手順に従い、リポジトリ全体の機密情報スキャンを実行すること。

- 🔴 CRITICAL または 🟠 HIGH の指摘が見つかった場合はコミットを中断し、ユーザーに報告して指示を仰ぐこと
- 🟡 MEDIUM 以下の指摘のみの場合は報告を行うが、ユーザーの判断でコミットを継続してよい

---

## フェーズ11: 要件充足ループ

全テストがパスし、セキュリティチェックが完了した後、要件リストと突き合わせて未充足の要件がないか確認する。

未充足の要件がある場合はフェーズ7（実装）に戻り、修正 → テスト → レビュー → セキュリティチェックのサブループを回す。ループの各周回で不明点があればユーザーに質問すること。

全ての要件を満たしたら次のフェーズに進む。

---

## フェーズ12: コミット・報告

```bash
git add <該当ファイル>
git commit -m "{日本語のコミットメッセージ}"
git push
```

コミット後、以下の内容をユーザーに報告する:

- 実施した内容の概要
- 追加したファイルのリスト
- 変更したファイルのリスト
- （必要な場合）動作確認方法や注意点
