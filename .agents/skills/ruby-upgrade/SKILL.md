Rubyのメジャーバージョンアップ（例: 3.4.x → 4.0.x）を実施する場合、このスキルの手順に従うこと。

## ブランチ運用フロー

コードを修正する際は、以下のフローで作業すること。

1. **ブランチ作成**: 作業開始前に `main` ブランチからトピックブランチを作成する (`git checkout -b <ブランチ名>`)。
2. **実装〜テスト〜コードレビュー〜セキュリティチェック**: 以下の各フェーズを実行する。コミットは修正の単位で行う。
3. **プッシュ**: 全フェーズ完了後、`git push origin <ブランチ名>` でリモートにプッシュする。
4. **PR作成**: `gh pr create --base main --head <ブランチ名> --title "..." --body "..."` でプルリクエストを作成する。
5. **レビュー依頼**: PRのURLをユーザーに伝え、レビューを依頼する。ユーザーがGitHub上でレビューし、マージする。
6. **マージ確認と後片付け**: ユーザーから「マージした」旨のメッセージを受けたら、`main` ブランチに切り替え、`git pull origin main` してマージを確認し、開発用ブランチを削除する (`git branch -d <ブランチ名>`)。

## ワークフロー概要

```
リサーチ（破壊的変更・影響調査）
  ↓
調査結果を提示 → ユーザーに続行/中止の確認
  ↓
修正箇所のリストアップ
  ↓
.バージョン確認（rbenv install --list）
  ↓
.ruby-version 書き換え
  ↓
（必要に応じてRubyインストールをユーザーに依頼）
  ↓
変更ループ: 1ファイルずつ修正 → test → rubocop → security → review
  ↓
最終確認（全テスト・全静的解析）
  ↓
コミット・報告
```

## フェーズ1: リサーチ

### 1.1 現在のバージョンとアップグレード先の決定

以下のファイルから現在のバージョンを確認する。

- `.ruby-version`
- `Dockerfile` の `ARG RUBY_VERSION`
- `Gemfile.lock` の bundler バージョン（間接的に）

アップグレード先のバージョン（例: 4.0.5）を決める。最新の安定版を選ぶこと。

### 1.2 バージョン間の破壊的変更のリサーチ

Ruby 公式のリリースノート (`ruby-lang.org`) を確認し、以下の分類で変更をリストアップする。

- **🔴 破壊的変更**: 削除されたメソッド・クラス・ライブラリ、互換性のない動作変更
  - 例: `Process::Status#&` / `#>>` の削除, `Ractor.yield` / `Ractor#take` の削除, `CGI` ライブラリのデフォルトgemからの削除
- **🟡 影響中程度の変更**: 非推奨化、デフォルト動作の変更、バックトレース形式の変更
  - 例: `ObjectSpace._id2ref` の非推奨化, Net::HTTP の Content-Type 自動設定廃止, バックトレース内の `<internal>` フレーム非表示
- **🟢 新機能**: 追加された機能（影響は通常ないが、注意点があれば記録）

### 1.3 現在のRailsバージョンとの整合性調査

- Rails の `required_ruby_version`（gemspec）を確認する
- アップグレード先のRubyバージョンがその条件を満たすか確認する
- Rails本体が削除されたAPIを使っていないか確認する（例えば `Process::Status#&` など）

### 1.4 現在使用しているgemとの整合性調査

以下の観点で全gemを調査する。

- 各gemの `required_ruby_version`（RubyGems.org または gemspecで確認）
- アップグレード先Rubyで削除されたAPIを使用していないか
- ネイティブ拡張gem（nokogiri, sqlite3, pg, commonmarker, thruster, bootsnap等）のRuby新バージョン対応状況

#### 調査結果の記録

以下のテーブル形式で記録する。

| gem | ロックバージョン | required_ruby_version | 問題 |
|-----|----------------|----------------------|------|
| rails | 8.1.3 | >= 3.2.0 | OK |
| nokogiri | 1.19.4 | >= 3.2 | OK |
| ... | ... | ... | ... |

### 1.5 アプリケーションコードの影響調査

以下のAPIがアプリケーションコードで使われていないか `grep` で確認する。

- `Process::Status` のビット演算
- `Ractor.yield`, `Ractor#take`, `Ractor#close_incoming`, `Ractor#close_outgoing`
- `ObjectSpace._id2ref`
- `IO.popen` の `|` プレフィックス
- `Kernel#open` の `|` プレフィックス
- `SortedSet`
- `CGI` ライブラリのフル機能
- Net::HTTP の Content-Type 自動設定に依存したコード

### 1.6 その他の確認項目

- Docker ベースイメージに Ruby 新バージョンの slim イメージが存在するか
- CI 環境に対応するバージョンがインストールされているか
- Bundler のバージョン要件（Ruby 4.0 には Bundler 4.x が同梱される）

## フェーズ2: ユーザーへの提示と確認

リサーチ結果を以下の形式でユーザーに提示する。

```
# Ruby {旧バージョン} → {新バージョン} アップグレード影響調査

## 🔴 破壊的変更（対応が必要なもの）
- ...

## 🟡 影響中程度の変更
- ...

## 🟢 問題なし（アプリに影響なし）
- ...

## Rails {現在のバージョン} との整合性
- ...

## Gem 整合性
- ブロックするgem: なし / （あればリスト）

## アプリコードへの影響
- ...

## 修正が必要なファイル
- ...

## 判断
このままバージョンアップを続けますか？
```

ユーザーの回答を待つ。

- ユーザーが「続ける」と答えた → フェーズ3へ進む
- ユーザーが「中止」と答えた → ここで終了

---

## フェーズ3: 修正箇所のリストアップ

影響調査結果を基に、修正が必要なファイルを全てリストアップする。

| ファイル | 修正内容 | 影響度 |
|---------|---------|--------|
| `.ruby-version` | `旧バージョン` → `新バージョン` | 必須 |
| `Dockerfile` | `ARG RUBY_VERSION` の書き換え | 必須 |
| `README.md` | スタック欄のRubyバージョン表記を更新 | 必須 |
| `AGENTS.md` | スタック欄のRubyバージョン表記を更新 | 必須 |
| `docs/*.md` | ドキュメント内の古いRubyバージョン表記を更新（`grep` で旧バージョンを検索して確認） | 必須 |
| `Gemfile` | 必要に応じて `gem "cgi"` 等の追加 | 状況による |
| その他アプリコード | 削除されたAPIの置き換え | 状況による |

各修正の影響度を「必須」「推奨」「状況による」で分類する。

## フェーズ4: バージョン確認と.ruby-versionの書き換え

### 4.1 インストール可能なバージョンの確認

```bash
rbenv install --list | grep '{アップグレード先のメジャーバージョン}'
```

アップグレード先のバージョンがリストに含まれているか確認する。

### 4.2 .ruby-version の書き換え

```bash
echo '{新バージョン}' > .ruby-version
```

### 4.3 バージョンがインストールされていない場合

```bash
rbenv install --list | grep '{アップグレード先のバージョン}'
```

リストにあるがインストールされていない場合 → ユーザーに `rbenv install {新バージョン}` の実行を促す。

エージェントが直接 `rbenv install` を実行してはならない。ユーザーの環境に影響を与える操作はユーザー自身に行ってもらうこと。

ユーザーから「インストール完了」の報告を受けたら次に進む。報告を受けていないのに先に進んではならない。

## フェーズ5: 変更ループ

**重要**: 修正箇所が多い場合でも、いっぺんに全てのファイルを修正してはならない。以下の順序で1ファイルまたは1論理グループずつ進めること。

### ループ1周の手順

1. **修正**: 1つのファイル（または密接に関連するファイル群）を修正する
2. **ユニットテスト実行**: `bin/rails test` でテストがパスすることを確認する
3. **Sorbet 型チェック**: `bundle exec srb tc` を実行し、型エラーがあれば修正する
4. **Rubocop**: `bin/rubocop` を実行し、指摘があれば修正する
5. **セキュリティテスト**: `bin/brakeman --no-pager` と `bin/bundler-audit` を実行する
5. **コードレビュー**: 別エージェント（`Task` ツール使用）によるレビューを依頼する
6. **問題があれば修正してループ**: 上記のいずれかで問題が見つかった場合は修正して再度ループする
7. **問題なければ次の修正へ**: 次のファイル/グループの修正に進む

このループを、フェーズ3でリストアップした全ての修正が完了するまで繰り返す。

### 実行順序の推奨

1. `.ruby-version` の書き換え（= 環境切り替え）
2. `Gemfile` の変更（gem追加・削除）→ `bundle install`
3. `Dockerfile` の変更
4. アプリケーションコードの修正（削除されたAPIの置き換え等）
5. テストコードの修正
6. 設定ファイルの修正
7. **ドキュメントのバージョン表記更新**: `README.md`, `AGENTS.md`, `docs/*.md`

---

## フェーズ6: 最終確認

全修正が完了したら、以下のコマンドを全て実行し、すべてがパスすることを確認する。

```bash
# 全テスト
bin/rails test

# システムテスト（必要な場合）
bin/rails test:system

# 型チェック
bundle exec srb tc

# Rubocop
bin/rubocop

# セキュリティスキャン
bin/brakeman --no-pager
bin/bundler-audit
bin/importmap audit

# 確認
ruby --version

# ドキュメント内の古いバージョン表記が残っていないか確認（該当がなければ出力なし）
grep -rn '旧バージョン（ドット省略なしの完全なバージョン番号）' README.md AGENTS.md docs/ 2>/dev/null || echo "ドキュメントのバージョン表記に問題なし"
```

---

## フェーズ7: セキュリティチェック

`git commit` を実行する前に必ず `.agents/skills/security-check/SKILL.md` の手順に従い、リポジトリ全体の機密情報スキャンを実行すること。

- 🔴 CRITICAL または 🟠 HIGH の指摘が見つかった場合はコミットを中断し、ユーザーに報告して指示を仰ぐこと
- 🟡 MEDIUM 以下の指摘のみの場合は報告を行うが、ユーザーの判断でコミットを継続してよい

---

## フェーズ8: コミット・報告

```bash
git add <該当ファイル>
git commit -m "{日本語のコミットメッセージ}"
```

コミット後、以下の内容をユーザーに報告する:

- アップグレード元・先のバージョン
- 変更したファイルのリスト
- テスト・静的解析の結果（すべてパスしたこと）
- デプロイ時の注意点（ベースイメージ変更の要否等）
- 可能であれば、軽く動作確認（`bin/rails s` で起動確認等）をしておくこと
