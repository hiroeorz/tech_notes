---

不具合修正の依頼を受けた場合、このスキルの手順に従うこと。

**各フェーズにおいて、不具合の再現手順や期待動作・実際の動作が不明瞭な場合は、その時点でユーザーに確認し、明確にしてから次に進むこと。** 推測で修正すると原因誤認や手戻りが発生するため、不明点は必ず確認する。

## ワークフロー概要

以下の各フェーズを順に実行する。**各フェーズの成果物がまとまった時点でユーザーに確認を求め、了承を得てから次のフェーズに進むこと。** 修正が完全でない場合は実装フェーズに戻ってループする。

```
不具合報告の内容理解
  ↓
現状調査（コード・直近変更・依存ライブラリ）
  ↓
原因の特定
  ↓
修正方針の提案
  ↓
再現テストの作成（可能な場合）
  ↓
コード修正
  ↓
Rubocop + 全テスト実行
  ↓
コードレビュー（別エージェント依頼含む）
  ↓
セキュリティチェック
  ↓
条件を満たすまでループ
  ↓
コミット・報告
```

---

## フェーズ1: 不具合報告の内容理解

報告された不具合について以下の項目を整理する。不明点があればユーザーに確認すること。

- **概要**: 何が起きているか（期待動作と実際の動作の差分）
- **再現手順**: どう操作すると発生するか（環境・画面・URL・入力値・操作順序）
- **影響範囲**: どの画面・機能・ユーザーに影響があるか
- **緊急度**: どの程度早急な対応が必要か（本番停止・一部機能不可・表示のみ等）
- **該当バージョン**: 再現した環境（本番 / ステージング / ローカル、コミットハッシュ）

成果物: 上記を箇条書きでまとめたもの（ユーザーに確認すること）。特に再現手順が不明な場合は必ず確認してから次に進むこと。

---

## フェーズ2: 現状調査

### 2.1 コード調査

- 不具合に関連するファイルを特定する（該当画面のビュー、コントローラ、モデル、ヘルパー、JavaScript、CSS）
- 関連ファイルのコードを読み、ロジックを追跡する
- エラーログや例外が発生している場合は、該当スタックトレースから原因箇所を特定する

### 2.2 直近の変更調査

`git log` で直近の変更を確認し、不具合がいつから発生している可能性があるかを特定する。

```bash
# 直近のコミット履歴
git log --oneline -20

# 特定のファイルに絞った変更履歴
git log --oneline -10 -- <関連ファイル>
```

不具合の発生タイミングとコミット時期が関連しそうな場合、`git diff` や `git show` で該当コミットの変更内容を確認する。

必要に応じて `git bisect` で原因コミットを特定する。

### 2.3 依存ライブラリの調査

- 不具合に関係しそうなGemやJavaScriptライブラリを洗い出す
- 直近でバージョンアップされたライブラリがないか `Gemfile.lock` や `package.json`（存在する場合）を確認する
- 該当ライブラリのGitHubリポジトリで、最近のリリースや報告されている類似のIssue / Pull Requestを確認する
- 既知のバグや互換性問題が報告されていないか調査する

```bash
# Gemfile.lock から特定Gemのバージョン履歴を確認
grep -A 1 '<gem-name>' Gemfile.lock

# git log で Gemfile.lock の変更を確認
git log --oneline -- Gemfile.lock
```

成果物: 調査結果の整理（ファイルパス・コミットハッシュ・ライブラリバージョン・関連Issue URLを明記）

---

## フェーズ3: 原因の特定

調査結果を基に、不具合の原因を特定する。

- **直接原因**: どのコードが期待と異なる動作を引き起こしているか
- **根本原因**: なぜそのコードが書かれたか（設計の見落とし / 想定漏れ / 結合ミス / ケアレスミス等）
- **条件**: どのような条件下で発現するか（特定のデータ状態・画面幅・権限・ブラウザ等）

特定した原因をユーザーに説明し、修正方針の提案に進んで良いか確認すること。

成果物: 原因の説明（直接原因 + 根本原因 + 発現条件）

---

## フェーズ4: 修正方針の提案

原因を踏まえ、修正の方針をユーザーに提案する。

- **修正方法**: どのファイルをどう修正するか
- **影響範囲**: 修正による他の機能への影響
- **代替案（ある場合）**: 複数の修正方法が考えられる場合、それぞれのトレードオフを説明する
- **テスト方針**: どのようなテストで再現および修正を確認するか

成果物: 修正方針の説明（ユーザー確認済み）。確信が持てない場合は複数案を提示し、ユーザーに選んでもらうこと。
修正内容が自明な場合はユーザーに確認せずに進んでよい。

---

## フェーズ5: 再現テストの作成

可能な場合、不具合を再現するテストを先に作成する（テスト駆動）。これにより修正の正当性を検証できるだけでなく、今後再び同じ不具合が再発することを防止する。

- 統合テスト（`test/integration/`）の場合は、該当のテストファイルに追加
- モデルテスト（`test/models/`）の場合は、該当のテストファイルに追加
- 既存のテストパターン（`test/integration/blog_flow_test.rb` 等）に従って記述する

テスト作成後、**修正前の状態でテストが失敗することを確認** すること（これによりテストが不具合を正しく捉えていることを確認する）。

例：
```bash
bin/rails test test/integration/blog_flow_test.rb -n test_name
```

成果物: 追加したテストコード。修正前にテストが失敗することの確認結果。

---

## フェーズ6: コード修正

修正方針に従ってコードを修正する。

- 修正は最小限にとどめ、不要な変更を加えない
- 既存のコーディングパターンに従う
- 修正後に `bin/rails test` で全テストがパスすることを確認する
- エッジケース（空データ・nil・権限不足等）を考慮する

---

## フェーズ7: 静的解析・テスト

### 7.1 Rubocop

```bash
bin/rubocop
```

指摘がある場合は修正すること。

### 7.2 全テスト実行

```bash
bin/rails test
```

既存テストが全てパスすることを確認する。追加した再現テストがパスすることも確認する。

### 7.3 システムテスト（必要な場合のみ）

```bash
bin/rails test:system
```

ブラウザ駆動のテストが必要な修正の場合のみ実行する。

---

## フェーズ8: コードレビュー（別エージェント依頼含む）

### 8.1 自己レビュー

以下の観点で自身の修正をレビューする。

- **修正の正確性**: 原因に対して正しい修正が行われているか
- **エッジケース**: 境界値・異常値・nil・空データで破綻しないか
- **エラーハンドリング**: 想定外の入力や状態で適切に動作するか
- **N+1**: データベースクエリにN+1が発生していないか
- **セキュリティ**: SQLインジェクション、XSS、CSRFの対策ができているか
- **既存コードとの一貫性**: 同じ問題に対して同じパターンで解決しているか
- **CSSクラス名**: 既存の命名規則に従っているか
- **JavaScript**: Stimulus コントローラの規則に従っているか

### 8.2 別エージェントへのレビュー依頼

Task tool を使用して別エージェントにコードレビューを依頼する。依頼時は以下の情報を含めること。

```markdown
依頼内容: 不具合修正のコードレビュー
修正ファイル: <ファイルパスのリスト>
修正内容: <修正の概要>
確認観点:
1. 修正が原因を正しく解決しているか
2. エッジケースや副作用の見落としがないか
3. 既存コードのパターンと一貫性があるか
4. 改善すべき箇所があるか
```

レビュー結果を受け取ったら、指摘事項を確認し、対応が必要な場合はフェーズ6（コード修正）に戻る。

---

## フェーズ9: セキュリティチェック

`git commit` を実行する前に必ず `.agents/skills/security-check/SKILL.md` の手順に従い、リポジトリ全体の機密情報スキャンを実行すること。

- 🔴 CRITICAL または 🟠 HIGH の指摘が見つかった場合はコミットを中断し、ユーザーに報告して指示を仰ぐこと
- 🟡 MEDIUM 以下の指摘のみの場合は報告を行うが、ユーザーの判断でコミットを継続してよい

---

## フェーズ10: 修正完了ループ

全テストがパスし、コードレビュー・セキュリティチェックが完了した後、不具合報告の内容と突き合わせて修正が完了しているか確認する。

未完了の場合はフェーズ6（コード修正）に戻り、修正 → テスト → レビュー → セキュリティチェックのサブループを回す。ループの各周回で不明点があればユーザーに質問すること。

---

## フェーズ11: コミット・報告

```bash
git add <該当ファイル>
git commit -m "{日本語のコミットメッセージ}"
git push
```

コミットメッセージは以下の形式を推奨する:

```
fix: 不具合の概要（原因と修正方法を簡潔に）

- 原因: xxx
- 修正: xxx
```

コミット後、以下の内容をユーザーに報告する:

- 不具合の原因
- 実施した修正内容
- 変更したファイルのリスト
- （必要な場合）動作確認方法や注意点
- コードレビューでの指摘事項と対応（あった場合）
