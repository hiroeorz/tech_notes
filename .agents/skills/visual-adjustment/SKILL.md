---
name: visual-adjustment
description: ブラウザでの画面の見た目を調整する手順を定義する。CSS調整、レイアウト修正、余白・色・フォント・テーマ調整など視覚的な変更を、デザイン要件と照合しながら行う。
---

画面の見た目（CSS / レイアウト / 余白 / 色 / フォント / テーマ / レスポンシブ崩れ等）の調整依頼を受けた場合、このスキルの手順に従うこと。

**各フェーズにおいて、依頼内容やデザインの意図で不明瞭な点があれば、その時点でユーザーに確認し、明確にしてから次に進むこと。**

## 作業フロー概要

```
依頼内容の把握
  ↓
デザイン要件の確認（docs/requirements.md, docs/images/*.png）
  ↓
現状コードの調査（CSS, ERB, Stimulus）
  ↓
修正方針の決定
  ↓
コード修正（CSS 変数 / セレクタ / レイアウト / ERB / JS）
  ↓
ブラウザ確認（browser-check スキルを参照）
  ↓  崩れや未達があれば修正に戻る
Rubocop + 全テスト実行
  ↓
サブエージェントによるセキュリティチェック
  ↓
修正完了・報告
```

---

## フェーズ1: 依頼内容の把握

以下の項目を整理する。不明点があればユーザーに確認すること。

- **調整箇所**: どの画面・コンポーネントのどこを調整するか
- **期待する見た目**: 具体的にどう変えたいか（色・サイズ・位置・余白・表示・非表示等）
- **参考画像**: スクリーンショットやデザインカンプがあればパスを確認する
- **現状の課題**: 現状で何が問題か（デザインイメージと異なる / 表示崩れ / 見づらい等）
- **関連デザイン要件**: `docs/requirements.md` の該当セクション番号

成果物: 上記を箇条書きでまとめたもの（ユーザーに確認すること）

---

## フェーズ2: デザイン要件の確認

### 2.1 要件書の確認

`docs/requirements.md` の該当セクションを読み、デザインの仕様・意図を把握する。

### 2.2 デザイン画像の確認

`docs/images/` または `docs/mobile_images/` に対象画面の参照画像がないか確認する。画像がある場合は以下の観点で現状との差異を特定する:

- 色味（アクセントカラー `var(--orange)` の使用箇所）
- フォント（`var(--display)` 明朝、`var(--sans)` ゴシック、`var(--mono)` 等幅）
- 余白・配置
- アイコンの有無と種類
- レスポンシブ時のレイアウト
- テーマ（light/dark）ごとの見た目

成果物: デザイン要件と現状の差異リスト

---

## フェーズ3: 現状コードの調査

### 3.1 CSSの調査

- 対象画面に関連する CSS を特定する（`app/assets/stylesheets/application.css` または該当ビューの inline style / `<style>` タグ）
- CSS カスタムプロパティ（`var(--paper)`, `var(--ink)`, `var(--orange)` 等）の使用状況を確認する
- テーマ切り替え（`.theme-dark`）対応の有無を確認する
- 既存のセレクタ命名パターンを確認する

### 3.2 ビュー・パーシャルの調査

- 対象画面の ERB ファイルを確認する
- HTML 構造と CSS クラスの付与状況を確認する
- `shared/_header.html.erb` や `shared/_sidebar.html.erb` など共通パーシャルの影響を確認する

### 3.3 Stimulus コントローラの調査

- 動的な表示制御（タブ切り替え、モーダル開閉、プレビュー等）に関わる JavaScript の動作を確認する
- `app/javascript/controllers/` 以下の該当コントローラを確認する

成果物: 関連ファイルのパス一覧と現状の構造メモ

---

## フェーズ4: 修正方針の決定

調査結果を基に修正方針を策定する。

- **修正方法**: どのファイルをどう変更するか（CSS 変数の変更、セレクタ追加、クラス名変更、HTML 構造変更等）
- **影響範囲**: 修正による他の画面・コンポーネントへの波及
- **代替案（ある場合）**: 複数の実現方法がある場合のトレードオフ
- **確認方法**: 修正後にどのように表示を確認するか

成果物: 修正方針の説明（ユーザー確認済み）。確信が持てない場合は複数案を提示し、ユーザーに選んでもらうこと。

---

## フェーズ5: コード修正

修正方針に従ってコードを修正する。

### 5.1 CSS 修正時の注意点

- `app/assets/stylesheets/application.css` に全 CSS が集約されているため、変更箇所が散らばらないようにする
- 色変更は可能な限り CSS カスタムプロパティ（`var(--orange)` 等）の値を変えるか、新しい変数を追加する。生の色コードを直接書かない
- テーマ（`.theme-dark`）対応が必要な場合は、必ず両方のテーマで崩れがないことを確認する
- レスポンシブ対応（メディアクエリ）が必要な箇所は、既存のブレークポイントに従う

### 5.2 ERB 修正時の注意点

- HTML 構造を変更する場合、既存の CSS セレクタが効かなくなる可能性があるため、影響範囲を確認する
- 新たに追加する CSS クラス名は既存の命名規則（kebab-case）に従う

### 5.3 テーマ対応

- ライトテーマの調整が完了したら、`.theme-dark` での見た目も確認し、調整が必要であれば同様に対応する

## フェーズ6: ブラウザ確認

`.opencode/skills/browser-check/SKILL.md`（browser-check スキル）の手順に従い、ブラウザでの表示確認を行う。

### 6.1 確認項目

- デスクトップ幅（1280x720）での表示崩れがないか
- モバイル幅（375x812）での表示崩れがないか
- 指定されたデザイン要件を満たしているか（`docs/images/*.png` との照合）
- コンソールエラーが出ていないか
- ネットワークエラーが出ていないか
- ライトテーマ / ダークテーマの両方で正常か
- フォームやインタラクション要素の見た目が適切か
- アクセシビリティ要素（ラベル、alt属性、ARIA、コントラスト比）が適切か

### 6.2 問題発見時の対応

確認で問題が見つかった場合はフェーズ5（コード修正）に戻る。以下の観点で原因を特定する:

- CSS セレクタの優先度・継承による効き漏れ
- メディアクエリの適用条件
- JavaScript による動的なクラス付与前の初期状態
- テーマ切り替え時の CSS 変数の未定義

## フェーズ7: 静的解析・テスト

### 7.1 Rubocop

```bash
bin/rubocop
```

### 7.2 全テスト

```bash
bin/rails test
```

CSS のみの変更であればテストへの影響は通常ないが、ERB・Stimulus・ヘルパーを変更した場合は必ず実行する。

### 7.3 サブエージェントによるセキュリティチェック

Rubocop と全テストが完了した後、Task tool で `security-auditor` サブエージェントを起動し、変更差分のセキュリティ監査を依頼する。依頼には変更ファイル、変更概要、確認済みのテスト結果を含め、特に以下を確認させる:

- ERB や Markdown 表示に XSS のリスクがないか
- フォームや Stimulus の変更で CSRF 対策や入力処理を弱めていないか
- 外部 URL、画像、アップロードを扱う変更に安全性の問題がないか
- 認証済みユーザー向け要素の表示条件や管理画面の認可を崩していないか
- 機密情報や個人情報を CSS、ERB、JavaScript、スクリーンショットへ混入していないか

Critical または High の指摘がある場合は完了扱いにせず、フェーズ5（コード修正）へ戻る。Medium 以下の指摘も内容を確認し、対応しない場合は理由を完了報告に記載する。

---

## フェーズ8: 修正完了・報告

以下の内容をユーザーに報告する:

- 実施した調整の内容
- 変更したファイルのリスト
- デザイン要件との対応（該当する `docs/requirements.md` セクション番号）
- 調整前後の比較（スクリーンショットがあれば添付）
- 残課題（ある場合）
- ブラウザ確認の結果
- サブエージェントによるセキュリティチェックの結果

---

## フェーズ9: コミット・プッシュ・PR作成

ユーザーからPR作成の指示を受けた場合、または作業完了時にユーザーがPR作成を希望する場合は、以下の手順で行う。

### 9.1 ブランチ確認

現在のブランチが `main` から派生したトピックブランチであることを確認する。

```bash
git branch --show-current
```

### 9.2 機密情報スキャン

コミット前に `.agents/skills/security-check/SKILL.md` の手順に従い、リポジトリ全体の機密情報スキャンを実行する。

- 🔴 CRITICAL または 🟠 HIGH の指摘がある場合はコミットを中断し、ユーザーに報告して指示を仰ぐ

### 9.3 コミット

```bash
git add <該当ファイル>
git commit -m "{日本語のコミットメッセージ}"
```

コミットメッセージは以下の形式を推奨する:

```
feat: 調整内容の概要

- 変更1
- 変更2
```

### 9.4 プッシュ

```bash
git push origin <ブランチ名>
```

### 9.5 PR作成

```bash
gh pr create --base main --head <ブランチ名> --title "..." --body "..."
```

PRの説明には以下の内容を含める:

- 調整の目的と概要
- 変更したファイルのリスト
- デザイン要件との対応（該当する要件セクション番号）
- 確認したチェック項目と結果
- 残課題（ある場合）
- スクリーンショット（あれば）

### 9.6 報告

作成したPRのURLをユーザーに伝え、レビューを依頼する。
