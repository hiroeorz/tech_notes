name: rails-upgrade
description: Railsバージョンアップの全工程を定義する。リサーチ・影響分析・Gemfile更新・コード修正・テスト・静的解析・セキュリティチェックまでをカバーする。
---

Railsのバージョンアップ（例: 8.1.x → 8.2.x または 9.0.x）を実施する場合、このスキルの手順に従うこと。

## ワークフロー概要

```
リサーチ（リリースノート・破壊的変更・影響調査）
  ↓
調査結果を提示 → ユーザーに続行/中止の確認
  ↓
修正箇所のリストアップ
  ↓
Gemfile のバージョン制約更新
  ↓
bundle update rails
  ↓
rails app:update 実行（コンフリクト対応）
  ↓
変更ループ: 1ファイルずつ修正 → test → rubocop → security → review
  ↓
最終確認（全テスト・全静的解析）
  ↓
コミット・報告
```

## フェーズ1: リサーチ

### 1.1 現在のバージョンとアップグレード先の決定

以下のファイルから現在のバージョンを確認する。

- `Gemfile` の `gem "rails", "~> X.Y.Z"`
- `Gemfile.lock` の `rails (X.Y.Z)`

アップグレード先のバージョンを決める。以下の方針で選択する。

- **マイナーバージョンアップ**（例: 8.1 → 8.2）: メジャーバージョン内の最新安定版。破壊的変更が少ない
- **メジャーバージョンアップ**（例: 8 → 9）: 大規模な変更が含まれる可能性が高い。マイグレーションガイドを必ず確認する

### 1.2 リリースノートの確認

Railsガイドのリリースノートを確認する。

- アップグレードガイド: `https://guides.rubyonrails.org/upgrading_ruby_on_rails.html`
- バージョン別リリースノート: `https://guides.rubyonrails.org/X_Y_release_notes.html`
- Rails公式ブログのリリース記事

以下の分類で変更をリストアップする。

- **🔴 破壊的変更**: 削除されたAPI、非推奨機能の削除、デフォルト動作の変更
- **🟡 影響中程度の変更**: 新たに非推奨になった機能、設定変更の推奨
- **🟢 新機能**: 追加された機能

### 1.3 使用gemとの整合性調査

全gemについて、新しいRailsバージョンで動作するか確認する。

- Gemfile.lock の全gemについて、`required_ruby_version` だけでなく Rails 本体への依存関係も確認する
- Rails engine として動作するgem（Devise, ActiveAdmin, Ransack 等）は特に注意する
- Solid Cache / Queue / Cable 等、Rails本体と密接に関連するgemの対応バージョンを確認する

#### 調査結果の記録

以下のテーブル形式で記録する。

| gem | ロックバージョン | Rails新バージョン対応 | 対応が必要な場合の新しいバージョン |
|-----|----------------|---------------------|--------------------------------|
| solid_cache | 1.0.10 | 未確認 | 確認中 |
| ... | ... | ... | ... |

### 1.4 非推奨警告の確認

現在のコードベースで、Railsの非推奨警告（deprecation warnings）が出力されていないか確認する。

```bash
bin/rails log:clear
# アプリケーションを起動していくつかの画面を開くか、テストを実行
bin/rails test 2>&1 | grep -i "deprecat"
```

### 1.5 アプリケーションコードの影響調査

以下の観点でアプリケーションコードを調査する。

- 非推奨メソッド・古いAPIの使用
- 新しいデフォルト設定との互換性
- 削除されるライブラリへの依存
- データベースアダプタ（SQLite / PostgreSQL）の互換性

## フェーズ2: ユーザーへの提示と確認

リサーチ結果を以下の形式でユーザーに提示する。

```
# Rails {旧バージョン} → {新バージョン} アップグレード影響調査

## 🔴 破壊的変更（対応が必要なもの）
- ...

## 🟡 影響中程度の変更
- ...

## 🟢 新機能（利用できそうなものがあれば）
- ...

## Gem 整合性
- ブロックするgem: なし / （あればリストと対応方法）

## アプリコードへの影響
- ...

## 修正が必要なファイル
- ...

## 判断
このままバージョンアップを続けますか？
```

ユーザーの回答を待つ。

- ユーザーが「続ける」と答えた → フェーズ3へ進む
- ユーザーが「中止」と答えた → ここで終了

---

## フェーズ3: 修正箇所のリストアップ

影響調査結果を基に、修正が必要なファイルを全てリストアップする。

| ファイル | 修正内容 | 影響度 |
|---------|---------|--------|
| `Gemfile` | `gem "rails", "~> X.Y.Z"` の書き換え | 必須 |
| `Gemfile` | 関連gemのバージョン制約更新 | 状況による |
| `config/` 以下のファイル | `rails app:update` で生成される新しいデフォルト設定との差分対応 | 必須 |
| `README.md` | スタック欄のRailsバージョン表記を更新 | 必須 |
| `AGENTS.md` | スタック欄のRailsバージョン表記を更新 | 必須 |
| `docs/*.md` | ドキュメント内の古いRailsバージョン表記を更新（`grep` で旧バージョンを検索して確認） | 必須 |
| アプリケーションコード | 削除/非推奨APIの置き換え | 状況による |

## フェーズ4: Gemfileの更新

### 4.1 バージョン制約の書き換え

```ruby
# Gemfile
gem "rails", "~> X.Y.Z"
```

### 4.2 関連gemのバージョン確認

Railsと密接に関連するgem（solid_cache, solid_queue, solid_cable 等）のバージョン制約も必要に応じて緩和する（または新しい制約を確認する）。

### 4.3 bundle update

```bash
bundle update rails
```

依存関係の競合が発生した場合:

1. 競合しているgemを特定する
2. そのgemの新しいバージョンが公開されているか確認する
3. 新しいバージョンに更新するか、代替gemを検討する
4. 解決できない場合はユーザーに報告し判断を仰ぐ

## フェーズ5: rails app:update の実行

```bash
bin/rails app:update
```

このコマンドは新しいRailsバージョンのデフォルト設定ファイルを生成する。コンフリクトが発生した場合は以下の手順で対応する。

### 5.1 ファイルごとの差分確認

`rails app:update` が示す各ファイルの差分を確認する。

- `config/application.rb`
- `config/environments/development.rb`
- `config/environments/production.rb`
- `config/environments/test.rb`
- `config/initializers/` 以下のファイル
- `config/database.yml`
- `bin/` 以下のファイル
- `config/boot.rb`
- その他

### 5.2 コンフリクト解決方針

- **新しいデフォルト設定**: 新しいデフォルト設定を採用することを基本とする。ただし、アプリケーションの動作に影響する重大な変更（認証・セッション・CORS等）は、ユーザーに確認してから変更すること
- **削除された設定**: 古いバージョンの設定が削除された場合、新しい方法に移行する
- **非推奨警告を抑制する設定**: Rails 7.x → 8.x 等では `config.active_record.legacy_connection_handling` のような移行用設定が削除される場合がある。エラーが発生した場合は新しい方法に移行する

`config/` 以下の変更はアプリケーション全体の動作に影響するため、1ファイルずつ慎重に確認し、必要に応じてユーザーに確認を取ること。

## フェーズ6: 変更ループ

**重要**: 修正箇所が多い場合でも、いっぺんに全てのファイルを修正してはならない。1ファイルまたは1論理グループずつ、以下のループを回すこと。

### ループ1周の手順

1. **修正**: 1つのファイル（または密接に関連するファイル群）を修正する
2. **ユニットテスト実行**: `bin/rails test` でテストがパスすることを確認する
3. **Rubocop**: `bin/rubocop` を実行し、指摘があれば修正する
4. **セキュリティテスト**: `bin/brakeman --no-pager` と `bin/bundler-audit` を実行する
5. **コードレビュー**: 別エージェント（`Task` ツール使用）によるレビューを依頼する
6. **問題があれば修正してループ**: 上記のいずれかで問題が見つかった場合は修正して再度ループする
7. **問題なければ次の修正へ**: 次のファイル/グループの修正に進む

このループを、フェーズ3でリストアップした全ての修正が完了するまで繰り返す。

### 実行順序の推奨

1. `Gemfile` の変更 → `bundle update rails`
2. `rails app:update` で生成された新しい設定ファイルの適用
3. 初期化ファイル（`config/initializers/`）の修正
4. アプリケーションコードの修正（非推奨APIの置き換え等）
5. テストコードの修正
6. `Dockerfile` 等のインフラ関連ファイルの修正
7. **ドキュメントのバージョン表記更新**: `README.md`, `AGENTS.md`, `docs/*.md`

---

## フェーズ7: 最終確認

全修正が完了したら、以下のコマンドを全て実行し、すべてがパスすることを確認する。

```bash
# 全テスト
bin/rails test

# システムテスト（必要な場合）
bin/rails test:system

# Rubocop
bin/rubocop

# セキュリティスキャン
bin/brakeman --no-pager
bin/bundler-audit
bin/importmap audit

# Railsバージョン確認
bin/rails about

# ドキュメント内の古いバージョン表記が残っていないか確認（該当がなければ出力なし）
grep -rn '旧バージョン（ドット省略なしの完全なバージョン番号）' README.md AGENTS.md docs/ 2>/dev/null || echo "ドキュメントのバージョン表記に問題なし"
```

## フェーズ8: デグレーションチェック

テストだけでなく、実際にアプリケーションを起動して以下の観点で動作確認する。

```bash
# 開発サーバー起動確認
bin/rails server -p 3001 &
sleep 5
curl -s -o /dev/null -w "%{http_code}" http://localhost:3001/up
kill %1
```

確認項目:
- アプリケーションがエラーなく起動すること
- トップページが表示されること
- 管理画面にログインできること
- 投稿の一覧・詳細・作成・編集ができること
- Markdownレンダリングが正しく動作すること

---

## フェーズ9: セキュリティチェック

`git commit` を実行する前に必ず `.agents/skills/security-check/SKILL.md` の手順に従い、リポジトリ全体の機密情報スキャンを実行すること。

- 🔴 CRITICAL または 🟠 HIGH の指摘が見つかった場合はコミットを中断し、ユーザーに報告して指示を仰ぐこと
- 🟡 MEDIUM 以下の指摘のみの場合は報告を行うが、ユーザーの判断でコミットを継続してよい

---

## フェーズ10: コミット・報告

```bash
git add <該当ファイル>
git commit -m "{日本語のコミットメッセージ}"
```

コミット後、以下の内容をユーザーに報告する:

- アップグレード元・先のバージョン
- 変更したファイルのリスト
- テスト・静的解析の結果（すべてパスしたこと）
- デプロイ時の注意点（新しい設定、非推奨警告、DBマイグレーションの要否等）
- 可能であれば、軽く動作確認（`bin/rails s` で起動確認等）をしておくこと
